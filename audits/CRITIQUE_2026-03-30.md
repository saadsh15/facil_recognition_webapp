# Architectural Critique: FaceRec Real-Time Recognition System

---

## 🚨 Critical Failures (Drop Everything and Fix)

---

### 1. `detect_and_embed` Calls MTCNN Twice Per Frame — 2x Inference Cost
**File:** `recognition.py:44-57`

`mtcnn.detect(pil_image, landmarks=True)` and `mtcnn(pil_image)` are called sequentially on the same image. MTCNN runs three cascaded CNNs (P-Net, R-Net, O-Net) on every call. You are paying the full inference cost twice per frame. On CPU this is catastrophic — recognition latency is doubled for zero gain. The fix is to use `mtcnn(pil_image, return_prob=True)` in a single call, which returns both the cropped tensors AND the probabilities simultaneously.

```python
# WRONG — two full MTCNN passes:
boxes, probs, _ = mtcnn.detect(pil_image, landmarks=True)
face_tensors = mtcnn(pil_image)

# CORRECT — one pass:
face_tensors, probs = mtcnn(pil_image, return_prob=True)
# Derive boxes from face_tensors if needed, or use mtcnn.detect once only.
```

### 2. `_pending_captures` Is an Unbounded, Unprotected In-Memory Memory Leak
**File:** `app.py:16`

This global dict stores numpy embeddings indefinitely. There is no TTL, no max size, and no eviction. Every call to `/api/capture-frame` that is not followed by `/api/faces` (user closes modal, network error, browser crash) permanently leaks 2KB of heap. Over a long-running process with repeated use, this accumulates without bound.

Worse: this dict is mutated from Flask request threads (`_pending_captures[capture_id] = ...`) and read/deleted from other threads (`_pending_captures.pop(capture_id)`). Python's GIL protects individual bytecode ops but does not protect compound read-check-write patterns. Under concurrent requests, a double-pop race is possible.

**Fix:** Replace with a `threading.Lock`-protected dict and a background cleanup job that purges entries older than 5 minutes using a timestamp:
```python
import time, threading
_pending_captures: dict = {}
_captures_lock = threading.Lock()

# On insert:
with _captures_lock:
    _pending_captures[capture_id] = (embedding, thumbnail, time.monotonic())

# Cleanup thread — purge entries older than 300s:
def _purge_stale_captures():
    while True:
        time.sleep(60)
        cutoff = time.monotonic() - 300
        with _captures_lock:
            stale = [k for k, v in _pending_captures.items() if v[2] < cutoff]
            for k in stale:
                del _pending_captures[k]
```

### 3. `create_face` Will Throw `AttributeError` on Malformed Requests — Unhandled 500
**File:** `app.py:62`

`request.get_json()` returns `None` when the request body is missing, the `Content-Type` is not `application/json`, or the body is invalid JSON. The very next line `data.get("capture_id")` then throws `AttributeError: 'NoneType' object has no attribute 'get'`. Flask will return a raw 500 with a stack trace exposed in development mode. This is a data-exposure vulnerability in dev and a broken API contract in prod.

```python
# Fix:
data = request.get_json(silent=True)
if not data:
    return jsonify({"error": "Request body must be valid JSON"}), 400
```

### 4. N+1 Database Queries Inside the Recognition Thread — Per Detected Face, Per Cycle
**File:** `camera.py:68-77`

For every recognition cycle, the code opens one DB session and then executes one `find_match` query **per detected face**. With 5 faces in frame at 5 FPS, this is 25 queries/second against PostgreSQL from a background daemon thread — all sequential, all round-trips. The session is created inside the loop via `session_factory()` which pulls a connection from the pool on every iteration. This hammers the connection pool.

The correct approach is to batch all face embeddings into a single query using `UNNEST` or a temporary values CTE, returning the best match per input embedding in one round-trip:

```sql
WITH inputs(idx, emb) AS (
    VALUES (0, :emb0::vector), (1, :emb1::vector), ...
)
SELECT DISTINCT ON (i.idx) i.idx, f.id, f.name,
       f.embedding <=> i.emb AS dist
FROM inputs i
CROSS JOIN faces f
ORDER BY i.idx, dist;
```

### 5. `get_models()` Has a TOCTOU Race Condition — Not Thread-Safe
**File:** `recognition.py:12-23`

```python
if _mtcnn is None:
    _mtcnn = MTCNN(...)  # Non-atomic — two threads can both see None
```

This is a textbook check-then-act race. Two threads simultaneously observing `_mtcnn is None` will both initialize models, resulting in two `InceptionResnetV1` instances being loaded into VRAM/RAM simultaneously, then one being discarded. On systems with 4GB VRAM, this can cause an OOM crash during startup. The initialization is protected only by accident — `app.py` calls `get_models()` synchronously before starting threads. Any future code change breaks this fragile ordering.

**Fix:** Use `threading.Lock` with double-checked locking, or simply initialize at module import time (not lazily).

### 6. Flask Thread Pool Exhaustion via Unbounded MJPEG Stream Consumers
**File:** `camera.py:97-112`, `app.py:22-26`

`generate_mjpeg()` runs an infinite `while True` loop inside a Flask response generator. Flask's default development server uses `threaded=True` — each connected stream consumer holds a thread for its entire lifetime. There is no limit on concurrent stream connections. Five browser tabs = five threads permanently sleeping in `time.sleep(0.033)`. The default Flask development thread pool will exhaust at low concurrent user counts, blocking all other API requests.

Furthermore, when a browser disconnects mid-stream, there is no `GeneratorExit` handling in `generate_mjpeg()`. The generator will not be collected until Flask detects the broken pipe on the next `yield`. This can be seconds of delay.

**Fix:** Add `try/except GeneratorExit` and connection tracking. For any serious use, replace Flask's dev server with Gunicorn + a streaming-capable worker (e.g., `gevent`).

## ⚠️ Architectural Smells & Tech Debt

---

### 7. Circular Import Architecture — `camera.py` Deferred-Imports `recognition.py` at Runtime
**File:** `camera.py:52`, `camera.py:168`

`from recognition import detect_and_embed, find_match` and `from recognition import detect_and_embed` are hidden inside function bodies to avoid a circular import. This is a code smell indicating the module boundaries are wrong. `camera.py` does too much: it owns the capture loop, the annotation loop, the MJPEG generator, AND the registration capture logic. These are four separate responsibilities.

The proper separation is:
- `camera.py` — only frame capture (`CameraThread`, `FrameStore`)
- `stream.py` — MJPEG generator
- `pipeline.py` — annotation loop (reads frames, calls recognition, writes annotated frames)
- `registration.py` — single-frame capture for registration

---

### 8. Module-Level Side Effects in `camera.py` and `database.py` — Untestable
**Files:** `camera.py:1-17`, `database.py:1-12`

Both modules execute side effects at import time: `load_dotenv()` and `create_engine(DATABASE_URL)`. `create_engine` immediately establishes a database connection. Importing `database.py` in a test without a running PostgreSQL instance will crash. There is no way to inject a test database URL or mock the engine. The entire backend is untestable in isolation.

**Fix:** Use an application factory pattern. Move `engine` and `SessionLocal` creation inside a `create_app()` factory that accepts configuration. Pass the engine/session factory explicitly rather than importing module-level globals.

---

### 9. `_recognition_thread` Creates a New DB Session Every Recognition Cycle
**File:** `camera.py:64`

`session = session_factory()` is called inside the `while True` loop — once every `1/RECOGNITION_FPS` seconds. Each call checks out a connection from the pool, uses it, then returns it. At 5 FPS this is 300 pool checkouts/minute from a background thread with zero value-add. A single long-lived session per background thread is correct here, with reconnect logic on failure.

---

### 10. `find_match` Uses Raw String Serialization of a Float List for SQL Parameter
**File:** `recognition.py:66-75`

```python
{"emb": str(emb_list)}
```

`str([0.123, 0.456, ...])` produces Python's default list repr: `[0.123456789012345, ...]`. This is then cast in PostgreSQL as `CAST(:emb AS vector)`. Several failure modes exist:
- Python can produce `nan` or `inf` for degenerate embeddings, which PostgreSQL's vector parser rejects, crashing the query.
- Python's float `repr` has no guaranteed decimal precision contract.
- The correct approach is to use `pgvector`'s native parameter binding: `pgvector.sqlalchemy.Vector` accepts a Python list directly in SQLAlchemy parameterized queries without string conversion.

---

### 11. No Per-Person Multi-Embedding Support — Accuracy Degrades With Pose/Lighting
**File:** `models.py`, `recognition.py`

The schema enforces one embedding per database row, and there is no mechanism to register the same person from multiple angles. Real-world recognition accuracy for InceptionResnetV1 drops by 15-30% when the query face is in a different pose than the single registered embedding. The correct schema has a `persons` table (id, name) and a `face_embeddings` table (id, person_id, embedding, angle_hint), with recognition returning the best match across all embeddings for a person via `GROUP BY person_id`.

---

### 12. `thumbnail` Stored as Base64 TEXT — Bloated API Responses
**File:** `models.py:17`, `app.py:47`

A 120x120 JPEG thumbnail base64-encoded is ~8-15KB of text per row. `GET /api/faces` fetches ALL faces and includes the full thumbnail for every single one. With 100 registered faces, a single list call returns 800KB-1.5MB of response — for what the user perceives as a sidebar face list. Thumbnails should be served as separate `/api/faces/:id/thumbnail` endpoints, with the list endpoint returning only `{id, name, created_at}`.

---

### 13. IVFFlat Index Built on Empty Table at Startup — Will Never Be Useful
**File:** `database.py:15-19`

`CREATE INDEX IF NOT EXISTS faces_embedding_idx ON faces USING ivfflat (embedding vector_cosine_ops) WITH (lists = 100)` is executed at startup when the table is empty. An IVFFlat index built on 0 rows has no cluster centroids. PostgreSQL will not use this index until it's rebuilt after sufficient rows are present. Additionally, `lists=100` is a hardcoded magic number that is only appropriate for ~10,000 rows (rule of thumb: `lists ≈ sqrt(rows)`). For the expected size of this app (tens of faces), `lists=10` would be correct. For serious scale, the index parameters need to be re-evaluated and the index rebuilt with `REINDEX`.

## 🐢 Performance Bottlenecks

---

### 14. `FrameStore.get_raw()` and `get_annotated()` Copy Every Frame
**File:** `camera.py:33-44`

Every call to `get_raw()` and `get_annotated()` does `frame.copy()` under a lock. A 640x480 BGR frame is `640 × 480 × 3 = 921,600 bytes` (~900KB). The recognition thread calls `get_raw()` at 5 FPS = 4.5MB/s of unnecessary memcopy. The MJPEG generator calls `get_annotated()` at 30 FPS = 27MB/s of memcopy. The lock contention from these copies also serializes the camera writer against the MJPEG generator.

**Fix:** Use a double-buffer or lock-free slot approach. Store a `(frame, version)` tuple and use an atomic version counter. Readers compare version numbers; if unchanged, they skip re-reading. Use `numpy`'s view semantics carefully: only copy when handing data outside the lock scope.

---

### 15. `generate_mjpeg` Re-Encodes the Same Frame at 30 FPS Regardless of Change
**File:** `camera.py:99-112`

The MJPEG generator unconditionally encodes and yields a frame every 33ms. If the recognition thread (running at 5 FPS) hasn't updated the annotated frame, the generator re-encodes and re-sends the identical frame six times. Each `cv2.imencode(".jpg", frame)` call is a full JPEG compression of a 900KB raw bitmap — even at quality 80 this is CPU-intensive. Track a frame version counter and skip encoding/yielding when the annotated frame hasn't changed since the last yield.

---

### 16. `_recognition_thread` Allocates a New PIL Image Every Cycle
**File:** `camera.py:60`

`Image.fromarray(cv2.cvtColor(raw, cv2.COLOR_BGR2RGB))` creates two intermediate array allocations per cycle: one for `cvtColor` (900KB NumPy array) and one for the PIL Image wrapper. At 5 FPS this is 10 heap allocations per second of ~900KB each, placing sustained GC pressure. PyTorch's MTCNN accepts NumPy arrays directly; the PIL conversion is only needed for the specific `MTCNN.detect` API. Pre-allocate the RGB buffer and reuse it.

## 🔍 Edge Cases Ignored

---

### 17. Camera Fails to Open — Silent Hot Spin
**File:** `camera.py:50`

If `cv2.VideoCapture(CAMERA_INDEX)` fails (wrong index, device in use, USB disconnect), `cap.isOpened()` returns False and `cap.read()` immediately returns `(False, None)` on every call. The thread then spins: check `ret`, sleep 50ms, repeat — forever. No log message is emitted, no retry with backoff is attempted, no notification reaches the user. The stream endpoint returns stale/null frames indefinitely with no indication of the hardware failure.

---

### 18. MTCNN Returns `None` for `face_tensors` When `boxes` Is Not None
**File:** `recognition.py:47-48`

`mtcnn.detect()` and `mtcnn()` are called separately. It's possible (due to MTCNN's internal alignment rejection logic) for `mtcnn.detect()` to return non-None boxes while `mtcnn()` returns `None` face tensors (face detected but failed alignment/margin check). The code checks `if boxes is None or face_tensors is None` but the subsequent `zip(boxes, probs)` assumes both have the same length. If box count and face_tensor count diverge due to the alignment rejection, the `face_tensors[i]` indexing will silently produce mismatched embedding-to-box assignments — incorrect recognition results without any error.

---

### 19. `capture_single_face` Uses a Stale Frame During High-Motion Scenes
**File:** `camera.py:120-122`

`store.get_raw()` returns whatever the last captured frame was, which could be up to 33ms old. If the user is moving when they click "Capture", the returned frame might not contain a face (or might contain a blurred/partial face). The registration flow has a "Retake" button to handle this, but the error message says "No face detected" with no indication that the problem was motion blur or a momentarily empty frame. No retry mechanism exists.

---

### 20. pgvector Query Returns Wrong Result When `faces` Table Is Empty
**File:** `recognition.py:67-75`

When the `faces` table is empty, `session.execute(...).fetchone()` returns `None`. The code handles this correctly: `if row is None ... return None, "Unknown", None`. However, on the very first recognition cycle after startup (before any face is registered), this query hits the empty table via the IVFFlat index — and PostgreSQL may log a warning or plan a sequential scan warning. More importantly, the recognition thread runs this query at 5 FPS against an empty table, generating constant unnecessary DB traffic for a system that has no faces registered.

**Fix:** Cache a `_has_faces` flag that the registration endpoint sets to True on first insert. Skip the DB query entirely when False.

---

### 21. Frontend: `VideoStream` Goes Silent on Stream Error — No Recovery
**File:** `frontend/src/components/VideoStream.jsx:10-12`

The `onError` handler hides the `<img>` element permanently. If the Flask server restarts, the stream drops, the image hides, and **the only way to recover is a full page reload**. There is no reconnect logic, no visible error message, and no retry timer. The "LIVE" badge stays visible over a black void.

---

### 22. `FaceRegister` Has No Timeout on the Capture Request
**File:** `frontend/src/components/FaceRegister.jsx:14-25`

If Flask is slow (model inference takes >2s under load), the component sits in `stage === "capturing"` indefinitely with no timeout. Axios's default timeout is none. The user sees "Detecting face..." forever with no way to cancel except closing the modal (which the `handleClose` function does support, but only if they know to click Cancel during what looks like a loading state).

## 💡 The Architect's Mandate (How to do it right)

---

### Fix #1 — MTCNN Single-Pass
```python
def detect_and_embed(pil_image: Image.Image):
    mtcnn, _ = get_models()
    # Single MTCNN pass: returns cropped tensors + probabilities
    face_tensors, probs = mtcnn(pil_image, return_prob=True)
    if face_tensors is None:
        return []
    # Recover boxes via a separate detect only if box coords are needed
    boxes, _, _ = mtcnn.detect(pil_image, landmarks=True)
    results = []
    for i, (tensor, prob) in enumerate(zip(face_tensors, probs)):
        if prob < 0.90:
            continue
        results.append({
            "box": [int(b) for b in boxes[i]] if boxes is not None else None,
            "prob": float(prob),
            "embedding": embed(tensor),
        })
    return results
```
Actually — the definitive single-pass solution: call `mtcnn.detect()` once to get boxes/probs, then manually crop-and-resize the faces yourself using `extract_face()` from `facenet_pytorch.utils.detect_face`. Avoids the double-pass entirely.

---

### Fix #2 — Thread-Safe Pending Captures With TTL
See Critical Failure #2 above. Use `_captures_lock` and store `(embedding, thumbnail, timestamp)`. Add a daemon cleanup thread. Cap `_pending_captures` at 100 entries maximum to bound memory usage.

---

### Fix #3 — Batch pgvector Query for Multi-Face Frames
Replace per-face queries with a single UNNEST batch query returning one best match per input embedding. See Critical Failure #4 for the SQL template. This reduces N database round-trips to 1 per recognition cycle regardless of face count.

---

### Fix #4 — Application Factory + Dependency Injection
Refactor to an application factory:
```python
# database.py
def create_db(database_url: str):
    engine = create_engine(database_url, pool_size=10, max_overflow=20)
    return engine, sessionmaker(bind=engine)

# app.py
def create_app(config=None):
    app = Flask(__name__)
    cfg = config or os.environ
    engine, SessionLocal = create_db(cfg["DATABASE_URL"])
    # Inject SessionLocal into routes via app.config or context
    return app
```
This makes the app testable: pass a test database URL, get an isolated app instance.

### Fix #5 — pgvector Native Parameter Binding
```python
from pgvector.sqlalchemy import Vector
from sqlalchemy import select, func

# Use SQLAlchemy's typed parameter binding, not str()
stmt = (
    select(Face.id, Face.name,
           Face.embedding.cosine_distance(embedding.tolist()).label("dist"))
    .order_by("dist")
    .limit(1)
)
row = session.execute(stmt).fetchone()
```
This uses pgvector's SQLAlchemy integration correctly, handles type safety, and avoids float repr issues entirely.

### Fix #6 — Multi-Embedding Person Model
```sql
CREATE TABLE persons (
    id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    name VARCHAR(255) NOT NULL UNIQUE
);

CREATE TABLE face_embeddings (
    id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    person_id UUID NOT NULL REFERENCES persons(id) ON DELETE CASCADE,
    embedding vector(512) NOT NULL,
    thumbnail TEXT,
    created_at TIMESTAMPTZ DEFAULT NOW()
);

CREATE INDEX ON face_embeddings USING ivfflat (embedding vector_cosine_ops)
    WITH (lists = 10);  -- Tune to sqrt(expected_row_count)
```
Recognition: query top-K embeddings across all persons, vote by person_id, return the person with the most matches below threshold.

### Fix #7 — VideoStream Auto-Reconnect
```jsx
import { useState, useEffect, useRef } from "react";

export default function VideoStream() {
  const [status, setStatus] = useState("connecting");
  const imgRef = useRef(null);

  useEffect(() => {
    let retryTimer;
    function connect() {
      setStatus("connecting");
      if (imgRef.current) {
        imgRef.current.src = `/stream?t=${Date.now()}`; // cache-bust
      }
    }
    function onLoad() { setStatus("live"); }
    function onError() {
      setStatus("disconnected");
      retryTimer = setTimeout(connect, 3000); // retry in 3s
    }
    const img = imgRef.current;
    img?.addEventListener("load", onLoad);
    img?.addEventListener("error", onError);
    connect();
    return () => {
      clearTimeout(retryTimer);
      img?.removeEventListener("load", onLoad);
      img?.removeEventListener("error", onError);
    };
  }, []);

  return (
    <div className="relative w-full rounded-xl overflow-hidden bg-black shadow-2xl">
      <img ref={imgRef} alt="Live feed" className="w-full h-auto block" />
      {status !== "live" && (
        <div className="absolute inset-0 flex items-center justify-center text-white">
          {status === "disconnected" ? "Stream lost — reconnecting..." : "Connecting..."}
        </div>
      )}
    </div>
  );
}
```
